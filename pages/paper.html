Question 1
a) Define the following terms:
i. Database: A database is an organized collection of related data that is stored and accessed electronically. It represents some aspect of the real world and is designed for a specific purpose, allowing efficient data storage, retrieval, and management.
ii. Database Management System (DBMS): A DBMS is a software system that enables users to define, create, maintain, and control access to databases. It provides an interface between the database and end users or application programs, ensuring data security, integrity, and efficient data management.
iii. Program-Data Independence: This is the ability to modify the schema definition at one level without affecting the schema definition at the next higher level. It allows changes to the data structure without requiring modifications to application programs that use the data.
iv. Data Model: A data model is a collection of concepts that can be used to describe the structure of a database. It includes data types, relationships, and constraints that should hold on the data. Examples include the relational model, hierarchical model, and network model.

Question 2
a) Database Schema vs Database State:

Database Schema: The description of the database structure, including table definitions, data types, and constraints. It rarely changes and represents the database design.

Database State: The actual data stored in the database at a particular moment. It changes frequently as data is inserted, updated, or deleted.

b) Data Abstraction Concept:

Data abstraction uses three levels:

  • External Level (View Level): Users interact with customized views showing only relevant data
  • Conceptual Level: Describes what data is stored and relationships between data
  • Internal Level: Describes how data is physically stored

Example: A manager querying employee salaries sees a simple report interface (external level) without knowing that data is stored across multiple indexed files on disk (internal level).

c) Characteristics of Relations:
  • Ordering of Tuples: Tuples are not ordered; there is no "first" or "last" row
  • Ordering of Attributes: Attribute order in a relation is not significant
  • Atomic Values: Each attribute value must be atomic (indivisible)
  • Unique Tuples: No two tuples can be identical; each tuple is unique
  • NULL Values: Relations allow NULL values to represent unknown or inapplicable data
d) Foreign Keys Specification:

Foreign Keys:

  • ENROLL.Ssn references STUDENT.Ssn
  • ENROLL.Course_code references COURSE.Course_code
  • BOOK_ADOPTION.Course_code references COURSE.Course_code
  • BOOK_ADOPTION.Book_isbn references TEXT.Book_isbn

Assumptions:

  • Ssn is the primary key of STUDENT
  • Course_code is the primary key of COURSE
  • Book_isbn is the primary key of TEXT
  • (Ssn, Course_code, Quarter) is the primary key of ENROLL
  • (Course_code, Quarter) is the primary key of BOOK_ADOPTION
e) Integrity Constraints:

Entity Integrity Constraint:

  • No primary key value can be NULL
  • Ensures each tuple is uniquely identifiable
  • Important because primary keys are used to reference tuples; NULL values would make identification impossible

Referential Integrity Constraint:

  • A foreign key value must either match a primary key value in the referenced relation or be NULL
  • Ensures consistency across related tables
  • Important because it prevents orphaned records and maintains valid relationships between tables

Question 5
a) Types of Update Anomalies:

Three types: Insertion, Deletion, and Update Anomalies

1. Insertion Anomaly:

Inability to insert certain data without the presence of other data.

Example: If a relation stores both employee and department info, you cannot add a new department until an employee is assigned to it.

2. Deletion Anomaly:

Deletion of data causes unintended loss of other important data.

Example: If we delete the last employee in a department, we lose all information about that department.

b) Functional Dependency:

Functional Dependency: A constraint between two sets of attributes. If X â†’ Y, then the value of X uniquely determines the value of Y.

Full Functional Dependency: Y is fully functionally dependent on X if Y is functionally dependent on X and not on any proper subset of X.

Transitive Functional Dependency: If X â†’ Y and Y â†’ Z, then X â†’ Z is a transitive dependency (where Y is not a candidate key).

c) Steps to Convert 0NF to 3NF:

Step 1: 0NF to 1NF

  • Eliminate repeating groups
  • Ensure each attribute contains atomic values
  • Identify primary key

Step 2: 1NF to 2NF

  • Ensure relation is in 1NF
  • Remove partial dependencies
  • Create separate relations for sets of values that apply to multiple records

Step 3: 2NF to 3NF

  • Ensure relation is in 2NF
  • Remove transitive dependencies
  • Create separate relations for attributes that depend on non-key attributes
