Database Architecture

Database Schema

A database schema is the blueprint or logical structure of a database. It defines how the data is organized and how the relationships between different data elements are associated. Think of it as the architectural plan for a house before it's built.

The schema specifies the tables, the columns within each table, the data types for each column, and the constraints (like primary keys and foreign keys) that enforce data integrity. It provides a formal definition of the entire database. It's important to distinguish between the schema (the structure) and the database instance (the actual data stored at a specific moment).

Three-Level Schema Architecture

Also known as the ANSI-SPARC architecture, this model provides a framework for building databases that promote data independence. It divides the database into three distinct levels of abstraction:

  • External Level (View Level): This is the highest level of abstraction and is what the end-users see. It describes only a part of the entire database that is relevant to a specific user group. For example, the HR department might see employee salary information, while the sales department sees customer order details. There can be multiple external views for a single database.
  • Conceptual Level (Logical Level): This level provides a unified, logical view of the entire database for the community of users. It describes what data is stored and the relationships between the data, without considering the physical storage details. The Database Administrator (DBA) typically works at this level to define the structure of all tables, entities, attributes, and constraints.
  • Internal Level (Physical Level): This is the lowest level and describes how the data is physically stored on storage devices. It deals with low-level details like file organization, data structures (e.g., B+ trees for indexing), and access paths. This level is managed by the DBMS and is generally hidden from programmers and administrators.

Data Independence

Data independence is a key advantage of the three-level architecture. It is the ability to modify a schema at one level without affecting the schema at the next higher level. There are two types:

  • Logical Data Independence: This is the ability to change the conceptual schema without having to change the external schemas or application programs. For example, you could add a new column to a table or merge two tables. As long as the original data required by the external views is still accessible, those views and their applications will not be affected.
  • Physical Data Independence: This is the ability to change the internal schema without having to change the conceptual schema. For example, the DBA might decide to change the file organization method to improve performance, such as moving the data to a new storage device or creating new indexes. These changes are hidden from the conceptual and external levels.

DBMS Languages

DBMS languages are used to define, manipulate, and control the data in a database. The most common database language, SQL (Structured Query Language), can be divided into several sub-languages:

  • Data Definition Language (DDL): Used to define and manage the database structure or schema. DDL commands include `CREATE` (to create tables), `ALTER` (to modify the structure), and `DROP` (to delete tables).
  • Data Manipulation Language (DML): Used for accessing and manipulating the data itself. DML commands include `SELECT` (to retrieve data), `INSERT` (to add new data), `UPDATE` (to modify existing data), and `DELETE` (to remove data).
  • Data Control Language (DCL): Used to manage access rights and permissions for users. DCL commands include `GRANT` (to give permissions) and `REVOKE` (to take away permissions).
  • Transaction Control Language (TCL): Used to manage transactions within the database to ensure data integrity. TCL commands include `COMMIT` (to save the changes) and `ROLLBACK` (to undo the changes).

Client-Server Architecture

This is a computing model that structures applications into two main parts: the client and the server. In the context of a DBMS, this architecture allows multiple users (clients) to connect to a central database server over a network.

  • The Client is the front-end application that provides a user interface. It is responsible for accepting user input, sending requests to the server, and presenting the results to the user.
  • The Server is the back-end system where the DBMS software runs and the database is stored. It is responsible for processing the client's requests, performing the necessary data operations (like querying or updating), and sending the results back to the client.

This model centralizes data management, enhances security, and improves scalability and performance, as the heavy data processing is handled by a powerful server.
