📊 Database Scenarios & ER Diagrams

Real-world examples with Entity-Relationship models

Scenario 01

University Course Management System

A university needs to manage students, courses, and enrollments. Students can enroll in multiple courses, and each course can have multiple students. Each course is taught by one instructor, but an instructor can teach multiple courses.

📋 Requirements:

  • Track student information (ID, name, email, major)
  • Manage course details (code, name, credits)
  • Record enrollments with grades
  • Store instructor information (ID, name, department)

ER Diagram

📝 Key Points:

This is a Many-to-Many (M:N) relationship between Students and Courses, requiring a bridge table (ENROLLMENT). The relationship between Course and Instructor is One-to-Many (1:N).

Scenario 02

Library Management System

A library manages books, members, and book loans. Members can borrow multiple books at different times. Each book can be borrowed by different members over time, but only one member can have a specific book copy at any given time. The system tracks when books are borrowed and returned.

📋 Requirements:

  • Store book information (ISBN, title, author, available copies)
  • Manage member details (ID, name, contact, membership date)
  • Track loan records (loan date, return date, status)
  • Monitor overdue books

ER Diagram

📝 Key Points:

The LOAN table acts as a bridge entity with additional attributes (LoanDate, ReturnDate). This M:N relationship allows tracking of book borrowing history and current loan status.

Scenario 03

E-Commerce Order System

An online store needs to manage customers, products, and orders. Customers can place multiple orders, and each order can contain multiple products. Each product has a price and quantity in stock. The system must track order details including quantity ordered and item price at the time of purchase.

📋 Requirements:

  • Store customer information (ID, name, email, address)
  • Manage product catalog (ID, name, price, stock quantity)
  • Record orders with date and status
  • Track which products are in which orders with quantities and prices

ER Diagram

📝 Key Points:

Customer-Order is a 1:N relationship (one customer can have many orders). Order-Product is M:N, requiring the ORDER_ITEM bridge table. The UnitPrice in ORDER_ITEM preserves the price at purchase time, even if the product price changes later.

Scenario 04

Hospital Patient Management System

A hospital needs to track patients, doctors, and appointments. Patients can have multiple appointments with different doctors, and doctors can see multiple patients. Each appointment includes a date, time, and reason for visit. Doctors belong to specific departments.

📋 Requirements:

  • Store patient records (ID, name, DOB, contact, medical history)
  • Manage doctor information (ID, name, specialization, department)
  • Track appointments (date, time, reason, diagnosis)
  • Record department details

ER Diagram

📝 Key Points:

Patient-Doctor has an M:N relationship through APPOINTMENT. Doctor-Department is N:1 (many doctors belong to one department). The APPOINTMENT entity stores additional information about each visit.

Scenario 05

Social Media Platform

A social media platform allows users to create posts, comment on posts, and follow other users. Users can create multiple posts, and each post can have multiple comments. Users can follow many other users and be followed by many users (a many-to-many self-referencing relationship).

📋 Requirements:

  • Store user profiles (ID, username, email, bio, join date)
  • Manage posts (ID, content, timestamp, likes count)
  • Track comments on posts
  • Record follower-following relationships

ER Diagram

📝 Key Points:

This design includes a self-referencing M:N relationship (FOLLOWS) where users follow other users. User-Post is 1:N, Post-Comment is 1:N, and User-Comment is also 1:N (users write comments).
